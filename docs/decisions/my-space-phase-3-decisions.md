각 섹션 형식:
- **질문**: 답해야 할 핵심 결정
- **선택지**: A/B/C 옵션과 trade-off
- **권장**: 권장 옵션 + 이유
- **블로커 여부**: sprint 시작에 필수인가, 진행하며 결정 가능한가

답변은 본 파일에 직접 체크/수정해서 commit 하거나, 각 섹션 끝의 "결정"란에 적어주면 된다.

## 3.1 이미지 업로드 (레시피 커버 / 일기 첨부)

### 결정 1.1 — 저장 백엔드
**질문**: 이미지를 어디에 저장할까?

| 옵션 | 장점 | 단점 |
|---|---|---|
| A. 로컬 볼륨 (`./uploads`) | 단순, 추가 비용 0, 현 self-hosted 와 잘 맞음 | 백업/이전 책임이 사용자에게, 컨테이너 ↔ 호스트 권한 이슈 |
| B. 외부 S3 호환 (Cloudflare R2 등) | 확장성, CDN, 백업 자동 | 비용, 비밀키 관리, 외부 의존성 |
| C. PostgreSQL bytea 컬럼 | 트랜잭션 일관성, 백업 쉬움 | DB row 크기 폭증, 캐시 불가, 5MB × 다수 → 성능 |

**권장**: **A (로컬 볼륨)** — self-hosted 단일 사용자 가정, R2 같은 외부 서비스 도입은 over-engineering. `docker-compose.yml` 에 named volume 추가 + nginx 가 정적 서빙.

**블로커**: ✅ 필수 — 결정 없으면 업로드 라우트 작성 불가

- 3.3 (Space 다중 UI) 만 먼저 — 결정 부담 가장 작음 (3.1, 3.2, 3.3 의 권장값으로 충분)
- 나머지는 본 결정 매트릭스가 채워진 다음 진행
